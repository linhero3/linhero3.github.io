Maven 简单入门

发表于 2018-05-06 |

一. Maven简介

用maven来管理项目目前已是java web领域的共识。maven，一个开源项目管理工具，采用pom(Project Object Model)的概念，将所有配置信息定义在pom.xml中。
简单来说，maven有如下优点：

  • 简化项目依赖管理，Jar通过坐标即可引入，易于上手。
  • 统一的目录结构，统一的软件构建阶段。
  • 仓库保存构建，便于协同开发。
  • 多插件支持，拓展容易。

如下是maven项目结构：

|-rootdir #pom/子目录
|—-src
|——–main/java #java源码
|——–main/resources #资源文件
|——–test/java #test代码
|——–test/resources #test资源文件
|—-target #jar/ear包
|—-target/classes #编译后的文件(class)

maven 安装

maven安装比较简单，下载解压设置系统路径即可。如下是maven目录结构：

bin: maven可执行命令
boot: 类加载框架包
conf: maven配置文件settings.xml
lib: maven运行时java类库

maven默认本地仓库: user/.m2/repository
maven默认远程仓库地址: https://repo.maven.apache.org/maven2，配置在lib目录的maven-modle-builder-3.x.x.jar的pom文件中，该pom是项目的superpom。

<repositories>
    <repository>
    <id>central</id>
    <name>Central Repository</name>
    <url>https://repo.maven.apache.org/maven2</url>
    <layout>default</layout>
    <snapshots>
        <enabled>false</enabled>
    </snapshots>
    </repository>
</repositories>

maven基本命令

maven的命令跟项目的构建过程相关，完整的构建过程包括清理、编译、测试、打包、集成测试、验证、部署

清理： mvn clean 删除target目录
编译： mvn compile src/main/java目录的java源码编译成class，放在target目录下
测试： mvn test
打包： mvn package 生成压缩文件jar/war
安装： mvn install 将构建物上传到本地仓库
部署： mvn deploy 将压缩文件上传到私服

二、maven生命周期与插件

maven定义了三个生命周期，分别有如下意义：

  1. clean -清理项目
  2. default -构建项目
  3. site -生成项目站点

maven生命周期，参考：
http://maven.apache.org/guides/introduction/introduction-to-the-lifecycle.html#Lifecycle_Reference

maven每个生命周期独立存在，每个生命周期由一个phase list构成，各个phase有依赖关系，后面phase执行时，前面的phase会按顺序依次执行。maven生命周期的各个阶段，包含了项目构建时的真正步骤。

  • clean生命周期

    pre-clean ：执行清理前的工作；
    clean ：清理上一次构建生成的所有文件；
    post-clean ：执行清理后的工作

  • default生命周期
    default生命周期是最核心的，它包含了构建项目时真正需要执行的所有步骤。

    validate
    initialize
    generate-sources
    process-sources
    generate-resources
    process-resources ：复制和处理资源文件到target目录，准备打包；
    compile ：编译项目的源代码；
    process-classes
    generate-test-sources
    process-test-sources
    generate-test-resources
    process-test-resources
    test-compile ：编译测试源代码；
    process-test-classes
    test ：运行测试代码；
    prepare-package
    package ：打包成jar或者war或者其他格式的分发包；
    pre-integration-test
    integration-test
    post-integration-test
    verify
    install ：将打好的包安装到本地仓库，供其他项目使用；
    deploy ：将打好的包安装到远程仓库，供其他项目使用；

  • site生命周期

    pre-site
    site ：生成项目的站点文档；
    post-site
    site-deploy ：发布生成的站点文档

maven生命周期是抽象的，具体工作由maven插件来完成。maven为大部分的phase绑定了默认插件。maven的一个插件可以执行多个目标(goal)，maven的phase最终会绑定在maven plugin的一个目标(goal)上。
maven plugins: https://maven.apache.org/plugins/index.html

三、pom.xml

pom配置项比较多，关注如下几个配置

  • profiles
    profile元素可以为一个特殊的环境定义一个特殊的构建。比如 production profile来运行mvn install，需要在命令行中传人-Pproduction。

  • dependencyManagement
    dependencyManagement一般定义在父pom，用来统一依赖版本号，其本身的定义并不会引入依赖。子项目引入依赖后不用指定版本号，会统一使用父dependencyManagement的定义。

  • build
    项目的插件列表

  • modules
    项目模块聚合

四、 settings.xml

maven的配置不想只给一个项目使用，可以将配置写在settings文件里，应用于多个项目，分发给多个用户使用。settings.xml可以写入仓库位置、认证信息等。如下对settings.xml主要配置做一说明。

  • servers

用来下载和部署的仓库是用pom中的repositories和distributionManagement元素来定义的。但是某些配置例如username和password就不应该随着pom.xml来分配了。这种类型的信息应该保存在构建服务器中的settings.xml中。

<servers>
    <server>
        <id>nexus</id>
        <username>deploy</username>
        <password>deploy</password>
    </server>
</servers>
  • mirrors

mirrors是maven的镜像仓库，maven中央仓库在国外，为了提高访问速度，国内建立了一些镜像仓库，如阿里的镜像仓库http://maven.aliyun.com/nexus/content/groups/public。
官方镜像仓库是：

http://repo.maven.apache.org/maven2 hosted in the US
http://uk.maven.org/maven2 hosted in the UK.

<mirrors>
    <mirror>
        <id>central</id>
        <name>Central Repository</name>
        <mirrorOf>*</mirrorOf>
        <url>http://maven.aliyun.com/nexus/content/groups/public</url>
    </mirror>
</mirrors>

mirrorOf
通过mirrorOf，可以指定那些仓库通过镜像仓库访问，用来获取更快的访问速度，有如下匹配规则：

*:使用一个仓库，所有的构件都从该镜像仓库download。这个仓库需要包含所有的构件，或者会代理到一个仓库group。如果mirrorOf是*，即使配了多个repository，也不会从这些repository中下载构件。 repo,repo1 : repo or repo1
*,!repo1 : everything except repo1，意思是repo1还是通过repository配置的仓库去下载，其余都从mirror配置的镜像仓库下载。
mirrorOf详解： http://maven.apache.org/guides/mini/guide-mirror-settings.html

proxies
代理配置

profiles
settings.xml中的profile是pom.xml中的profile的简洁形式。它包含了激活(activation)，仓库(repositories)，插件仓库(pluginRepositories)和属性(properties)元素 如果settings中的profile被激活，那么它的值将重载POM或者profiles.xml中的任何相等ID的profiles。

  • repositories
    仓库是maven构建本地项目的远程仓库集合。当profile被激活，就会在该仓库寻找匹配的构件(jar)。

  • pluginRepositories
    类似repositories，不过一个是构件仓库，一个是插件仓库。

  • properties
    Maven的属性是值占位符，如果x是一个属性的话，那么它的值在pom中可以使用${x}来访问。

  • activeProfiles
    激活对应的profile(根据profile id匹配)。

五、 其他相关

版本号规范
Maven有发布版本（也称正式版本）与非发布版本（也称开发版本）的区别。
规范：发布版本由数字、字母、小数点和短横线组成，非发布版本由 -SNAPSHOT 后缀组成。

发布版本示例：1.0.0 1.1.0-RELEASE
非发布版本示例：1.0.0-SNAPSHOT Maven

在执行部署时可以自动识别版本规范，将正式版本部署到Releases仓库，非正式版本部署到Snapshots仓库。Releases仓库禁止重复部署相同坐标的Jar包。如果有新的版本要发布，必须升级版本号，否则会收到HTTP 400错误。

如果已经发布了jar包，现在这个包已经有代码修改，需要再次发布，那么必须升级版本号。可以通过修改 pom.xml 里的 version 节点，或者 maven deploy 命令里的 -Dversion 参数实现版本升级。推荐：使用 maven-release-plugin 插件来管理版本的自动升级。

maven项目的依赖、继承、组合

<dependency/>
<parent/>

依赖范围

<dependency>
        <groupId>javax.servlet</groupId>
        <artifactId>servlet-api</artifactId>
        <version>2.5</version>
        <scope>provided</scope>
</dependency>

compile:编译依赖，默认的依赖方式，在编译（编译项目和编译测试用例），运行测试用例，运行（项目实际运行）三个阶段都有效，典型有spring-core等jar。
test:测试依赖，只在编译测试用例和运行测试用例有效，典型地有JUnit。
provided:对于编译和测试有效，不会打包进发布包中，典型的例子为servlet-api,一般的web工程运行时都使用容器的servlet-api。
runtime:只在运行测试用例和实际运行时有效，典型地是jdbc驱动jar包。
system: 不从maven仓库获取该jar,而是通过systemPath指定该jar的路径。
import: 用于一个dependencyManagement对另一个dependencyManagement的继承。

依赖传递及冲突处理

  1. 依赖传递 A->B，B->C，则A会自动依赖C。

  2. 依赖规则

    路径长度一致，先定义先引入：A->B->X(1.0) A->D->X(2.0)=>引入X(1.0)
    最短路径引入：A->B->C->X(1.0) A->D->X(2.0)=>引入X(2.0)

  3. 依赖冲突处理 排除依赖 A->B->C(1.0) 不想使用C(1.0)，可在B中排除C(1.0)，并在A中引入C(2.0)

六、 Nexus 私服

nexus私服作为一种特殊的远程仓库为Maven提供仓库服务，当Maven需要下载构件的时候从私服请求，若私服上不存在，则从外部远程仓库下载，缓存到私服上后，再为maven的下载请求提供服务。
此外，一些无法从外部下载的构件也能从本地上传到私服供其他用户使用。构件搜索顺序：本地仓库、私服、中央仓库。
Nexus包括多种仓库概念，包括代理仓库（proxy）、宿主仓库（host）、仓库组（group）等，每一种仓库都提供了配置参数供用户定制。
目前Nexus支持maven2/npm等私有仓库。Nexus 私服搭建请搜索，简单来说就是下载、启动，然后在web页面做部分配置。
